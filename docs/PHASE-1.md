# 第一阶段开发指南

## 阶段目标

完成一个自己可以每天使用的文本收件箱。用户能够快速保存一段内容，系统能够可靠地生成结构化整理结果，并允许用户确认 AI 建议的待办。

建议周期为 5～7 个有效开发日。不要按照“把所有页面搭完”推进，而要按照纵向闭环推进。

## 开始之前要确定的三件事

1. 第一阶段只支持单用户。本地使用时可以暂不做登录，但所有表保留 `userId` 扩展位置。
2. 第一阶段只接受纯文本。链接先作为普通文本保存，不抓取网页正文。
3. AI 失败不能导致原始内容丢失。必须先写数据库，再触发 AI 处理。

## AI 输出契约

模型必须返回经过 Zod 校验的结构化数据，而不是让业务代码解析自然语言：

```ts
type AnalysisResult = {
  title: string;
  summary: string;
  contentType: "note" | "idea" | "reference" | "task" | "other";
  priority: "low" | "medium" | "high";
  tags: string[];
  tasks: Array<{
    title: string;
    description?: string;
    dueAt?: string;
    priority: "low" | "medium" | "high";
  }>;
};
```

日期不明确时不要猜测；`dueAt` 应为空。AI 提取出的任务初始状态为 `suggested`，必须由用户确认。

## 开发顺序

### 里程碑 0：初始化项目

- 创建 Next.js TypeScript 项目
- 配置 ESLint、Prettier 和环境变量校验
- 启动 PostgreSQL
- 接入 Drizzle/Prisma并创建首个迁移
- 建立 `.env.example`，禁止提交真实密钥
- 添加 `npm run lint`、`npm run test`、`npm run typecheck`

完成标准：应用和数据库都能启动，CI 风格的三个检查命令可以执行。

### 里程碑 1：先完成无 AI 收件箱

- `/capture`：一个文本框和保存按钮
- `POST /api/items`：保存原始文本
- `/inbox`：按时间倒序展示记录
- `/inbox/[id]`：查看原始内容
- 支持归档和收藏

完成标准：即使没有模型 API Key，DropMind 也能作为普通收件箱使用。

### 里程碑 2：加入 AI 整理

- 创建 `lib/ai` 模型适配层
- 编写内容分析 Prompt
- 使用 Zod 校验模型输出
- 保存标题、摘要、类型、优先级和标签
- 增加 `processing`、`ready`、`failed` 状态
- 失败时显示“重新处理”按钮
- 将调用耗时、模型和错误记录到 AIJob

推荐处理顺序：

```text
保存 rawContent
  → 返回 itemId
  → 调用 AI
  → 校验结构化结果
  → 事务写入分析结果、标签和建议任务
  → 更新状态
```

第一阶段可以在请求内直接完成 AI 调用，暂时不引入 Redis。接口应设计为以后可以替换为队列任务。

完成标准：针对同一批测试文本，输出格式稳定；模型失败时原始内容仍然存在且可以重试。

### 里程碑 3：待办确认

- `/tasks`：分别展示“待确认”和“进行中”
- 用户可以接受、修改或忽略建议任务
- 可以标记完成
- 任务详情可以跳回来源内容

完成标准：AI 不能直接创建正式待办；用户可以追溯每个任务从哪里提取出来。

### 里程碑 4：关键词搜索

- 搜索标题、摘要、原文和标签
- 支持状态、类型、优先级筛选
- 搜索结果显示命中原因和内容摘要
- 空状态和无结果状态清晰可见

第一阶段使用 PostgreSQL 文本搜索或 `ILIKE` 即可，不要提前接入向量数据库。

完成标准：用户可以在十秒内找到一周前保存的内容。

### 里程碑 5：打磨真实使用体验

- `Ctrl/Cmd + K` 打开快速投递
- 保存成功后立即清空输入框并给出反馈
- 加载、失败、重试和空状态完整
- 手机尺寸可以完成投递和查看
- 准备 15～20 条真实测试数据
- 连续使用七天，记录摩擦点

## 第一批 API

```text
POST   /api/items
GET    /api/items
GET    /api/items/:id
PATCH  /api/items/:id
POST   /api/items/:id/analyze
GET    /api/tasks
PATCH  /api/tasks/:id
GET    /api/search?q=...
```

不要把模型调用直接写在路由文件里。路由只负责参数校验、鉴权和调用业务服务。

## 第一批页面

```text
/capture      快速投递
/inbox        收件箱列表
/inbox/[id]   内容详情和 AI 处理结果
/tasks        建议任务与正式待办
/search       搜索和筛选
```

可以先让 `/` 重定向到 `/inbox`，暂时不做营销首页。

## 测试重点

- 空文本、超长文本和特殊字符
- 没有任务的内容
- 一段内容包含多个任务
- 模糊日期不能被 AI 编造为确定日期
- 模型超时、返回非 JSON 或字段缺失
- 重试不会重复创建标签和任务
- 删除或归档来源内容后，任务引用仍然一致

## 第一阶段完成定义

同时满足以下条件才算完成：

- 能保存并查看原始文本
- AI 可以生成经过校验的结构化整理结果
- AI 失败可见、可重试且不丢数据
- 建议任务需要人工确认
- 可以按关键词和标签找到历史内容
- 核心流程有自动化测试
- 连续实际使用七天，并形成下一阶段问题清单

## 现在从哪里开始

今天只做里程碑 0 和里程碑 1 的最短路径：

1. 初始化 Next.js。
2. 启动 PostgreSQL并连接成功。
3. 建立 `InboxItem` 表，只放 `id`、`rawContent`、`status` 和时间字段。
4. 完成 `/capture` 的文本保存。
5. 完成 `/inbox` 的列表展示。

先不要接 AI。当天结束时，应该已经拥有一个能保存真实内容的普通收件箱；第二天再在这个可靠底座上加入模型能力。
